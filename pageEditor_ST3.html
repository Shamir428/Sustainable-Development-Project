This page includes the technical implementation of the pages I created for the group coursework: feedback.html, team.html, and content_ST3.html.

Feedback Page

Technical Description


The Feedback page was built using the shared website structure which was the header and footer, as well embedding the global css. Inside the main part of the page, I organized it into an intro section, a programmes section, a ratings and reviews section, and then the feedback form. This was to create distinction between different parts of the page in terms of both visually and also purpose wise.

Programme section

In the programmes section, main features were the filter bar (which doesn't functionally work) and and the programme cards displaying different opportunities available to do for SDG 3. I implemented the filter bar by first creating a div container to add and group multiple elements together for it, as using the select form control element on its own (which creates a dropdown list) was not enough as form html elements don't have much css properties for visual styling. Inside the div container, I added the select element and label for it. As far as major CSS styling and techniques, I applied border and padding css properties to the select element, to display selected dropdown menu option more prominently and make it more readable, as by default, the selected option almost touches the elements border and is not visually prominent. I also applied background color and border css properties to the filter bar div. Initially this was done to make the select dropdown element look more like a filter bar, but then I realized, because this is a block element, it takes up the full available width, so the added background color and border css styling, created a neat but appealing visual distinction and transition between programme section intro and the programme cards. Lastly, this was also beneficial for the next part as well, because the programme cards content and information had to be scrollable, however, without clear visual indication of where specifically on the programme section is scrollable and not, users could have trouble trying to scroll through the content, worsening usability of website.

Programme card's grid layout

For the programme cards, I decided to use a div container first. This was because I wanted to arrange and style the cards in a consistent and even layout. For the SDG 3, I felt like I had a many options to talk about for opportunities you can do. Which is why I decided to make the div a grid container (display: grid) to be able to arrange elements inside div in rows and columns consistently, as displaying the cards horizontally would not fit and vertically would not look good, and using inline-block or other properties, would not be as predictable and consistent as using display grid. I defined the grid layout as three equal columns by using 1fr 1fr 1fr css value for the css property. To enable scrolling effect of the content inside grid (programme cards) I limit the max height of the grid to 55% of the view port, so that the content inside the programme section will occupy at most about half of the screen height, leaving any remaining content to end outside. So then I applied overflow: auto to grid, to contain the programme cards inside the grid and what ever exceeds the height limit becomes scrollable part of content.

Programme cards

The programme cards are implemented using an article tag, which visually visually looks and behaves exactly like a div but semantically, it is used to represent a self-contained piece of content, which a programme card fits in this context. Inside each article tags, I added a h3 heading for the programme title, to maintain clear semantic structure and hierarchy for the html, as I include a h2 section heading earlier, making this heading to be a subheading of the h2 heading. I've also added an image illustrating each of the programmes with appropriate alt text for accessibility for screen readers, paragraph and button in each article. For CSS styling, I first added background color for the card, to give the visual distinction of the cards compared to the programme section background. But to create the visual appearance of a card, I applied gap css property to the grid, so that there was horizontal spacing between the columns, indicating where now a card starts and ends (horizontally). Other techniques, I've used was for instance applying object-fit: cover declaration to the image inside of the card. This was because initially since all of the images' sizes were much bigger than the actual card, the images's aspect ratio was changed scale down to fit inside the card, making images look worse. With object-fit: cover, it kept the image aspect ratio the same but cut off the content was went over the image element box. Last major css techniques I used were, adding a hover effect to the button, by adding the pseudo-class :hover then applying css properties like background color and border radius with different values than normal button like.

Reviews summary section

The next section for ratings and reviews summary, I used div container overall rating display of the programmes with article elements again to create the review cards. I've followed a similar approach of visually creating the appearance of the card, styling the border, radius, adding gap property. But I wanted to create a clear visual distinction between the two cards, and why this card makes more sense to be seen as for reviews adding a different background color to the card, changing the positioning of the text from previously on the left side of the programme card to the middle of the review card, as text being displayed in the middle, usually represents less focus on informational text (as information text usually is long, therefore, for the sake of spacing, it's better to be left aligned) and more emphasis on the content rather than amount of information, which fits reviews more.

Feedback form

The feedback form I wrote is a full and basic complete code of a form with fieldsets, legends and form control elements. However I did not add any attributes in the form tag to let the form send or get data from anywhere. I also added novalidate attribute to prevent the browser's default validation of some inputs, to create my own. In order to make the form interactive such as adding data input validation upon submit printing success message if all inputs are correct and live remaining character counter for feedback message, I added id's with clear names to each form control input in the form. In order to print the potential error messages, I added span elements with no html text by default below each label. I used the CSS technique of using another pseudo-class :empty with class for error message spans to select the span if its text is empty and added display: none, to hide the element including the its vertical spacings it would've had, creating unintentional vertical space between label and input control if no error message is shown. Lastly for the live remaining character text, I added a p element below textarea element for feedback message and wrote "Remaining characters:" inside it but then also added a span element with an id to be later to display the amount of characters remaining dynamically.

JS for Live remaining characters counter

I first used the DOM function getElementById and the argument value of the id name of the span for remaining characters text on the document objet which represents the whole html page to find the span element's DOM object and stored in remainingCharsText variable. The variable now holds the reference to the DOM object which represents the remaining chars span, so I used const to create variable instead of let, as variables don't store values of objects but just reference, which for safety can stay fixed. I also created a const variable storing the referencing of the feedback's textarea input using same DOM method but used the id feedback to find the feedback textarea element. With two of the main elements' DOM objects stored as variables, I attached an event listener to the feedback textarea element with "input" event as the first parameter, which will listen to any input changes to the textarea element (whether it's adding or removing characters) and fires an event handler which is a function, in this case I reference and define as an anonymous function. This function will implement the counting logic for remaining characters of the feedback text. I first created a variable that takes the textarea's object's value which in this case is the value of the actual text message typed as object's have attributes and methods that you can use. To count remaining characters left, I created another variable to store the length of the feedback text message Then I subtracted the length and current amount of characters feedback message has from 250, as that was my intended max character count, and stored that in remainingChars variable. Lastly, using the remaining chars DOM element defined earlier, I updated the span's text content using the textContent property and setting it equals to the remaining characters variable So lastly, this will then compute every time, and update the live counter of remaining characters left.

JS for input validation and success message for form

First I created the variables for all of the form control and the different error message span elements' DOM objects I added this time another event listener, this time for submit event which the form can produce a submit button is clicked, hence I attached the event listener to the form DOM object. This time for the function argument, I referenced a named function. That function is called handleForm() and it will run and check each time the form gets submitted. The event handler function itself will not however not directly handle the logic of validating the different form inputs, as coursework specs say, form should handle both input validation and printing success messages, so this function will handle what type of action the form should make after submission. So first, by default, forms even without get or post method will refresh or redirect back to the top of the page after submission, but I want to either print all error messages from my own input validation logic or print a success message, so I first for the function definition add an argument with name event, which would during function call / during event handling take the passed argument value of the event object, which is an object which is created each time the specific event an event listener is listening to is fired. The event object has a method called preventDefault() which prevents the default action of what the trigger of this event does. Then lastly I will call the validateForm() function but also I want to handle the logic of printing success message, so I added an if statement around validateForm(), and if the value of that function call is true, it will print success message. Now lastly, as just revealed, the validateForm() will print a true or false boolean value based on user input. However, that is mainly for the if statement to check if all inputs are correct, to print success message. Because validateForm consists all if statements which check to an adequate extent if all of the different input values upon form submission is correct - is it not empty, follows the format, etc. I used several techniques such as using String methods and regex to validate the some of the input elements' values because String methods can give you multiple insights of a string like checking the length, if it contains certain specific characters and also provides ways to modify the strings (but it returns a new string instead) such as cleaning the input data before validating by for example removing trailing white space or removing acceptable symbols in the phone number input. Lastly a regex pattern is an object which defines a matching rule for text using a sequence of characters. In this case it was helpful because I was able to, partially for instance, add some input validation for the telephone number such as adding an if statement to check if the regex pattern of specific characters valid for a telephone number such as digits, +, etc. is valid or not valid for the telephone string.

Accessibility

For accessibility, I tried to make the feedback page clear and understandable by using proper labels for the form fields, fieldsets and legends to group related inputs, and alt text for all images in the programme cards. Error messages are shown close to the relevant fields, so it is obvious for the readers what input is wrong. I also added an aria-live="polite" attribute for the form status message so screen reader users can be informed when the confirmation text appears after a successful submit. The page also uses a clear heading structure and seperate sections, which helps with readability and navigation.

Team Page

Technical Description

The Team page was built using the same shared website structure with the same header, navigation menu and footer. The main content of the team page contains the page heading and short intro paragraph text. Then it features a section called team-members where I implemented article elements again for the cards of each of the team members. As far as the CSS techniques used, this time I wanted the card to be much more visually prominent, as the card's visual aesthetics was part of the showcase of team members, as for the feedback page according to the specs I didn't have to write as much as in the feedback page. So because we only had 3 members, I knew that it would be most efficient to display the cards in a row. And if I'm just going to deal with rows, I will use a flex container to display the cards inside it, which is the section with class called team-members. For CSS techniques, since the specs said to make the team cards more interactive and have visual effects on hover such as revealing detailed member details only on hover, I used the css property transform and translateY() css value to extend the team member card's to end deeper. I did this so that when the card is hovered over, using pseudo-class and those css properties, I can add the area of where the detailed members description is shown. For the detailed members description, I first set their default state (when it's not hovered) to be invisible using opacity, have a max height of 0, because otherwise they would just behave as any normal text and taking up additional vertical space of the card , so before it's hovered and revealed, there would be unnecessary deadspace. And to prevent the member details text by default going over the card, since the card only expands / extends lower if it's hovered, I applied overflow: hidden; to the member details text which hides any content inside an element in this case the card from going over it, because once the card extends, that's where the content of the members details text can be shown. So then lastly I added a hover and focus pseudo-classess on the css selector of the team member card, when the card is hovered or clicked the member details text, smoothly is revealed and finally shown by adding the opacity: 1 declaration and a transition css property with the value of 0.3 seconds which means when one of the team member cards for instance get hovered, first the card extends then the detailed description gets slowly revealed (just 0.3 seconds). I also added the transition css property to the cards when they are hovered, so that the effect of the card extending deeper is more smooth. Lastly I've used few more CSS techniques such as scaling the size of the image by using transform: scale() css declaration and also box shadow css property to make the cards more visually prominent as I wanted.

Accessibility

For accessibility, I included alt text for each team member image and kept the text structure clear with headings and short descriptions. I also made each team card keyboard-focusable by adding tabindex="0", so the cards can be usable by keyboard input as well. This helps make the page more accessible and easier to navigate in different ways.

Content Page

Technical Description

My content page focuses on preventive health. The main section first begins with a hero section introducing the topic, followed by an internal navigation section labelled and its label "on this page", and then multiple content sections with unique id values. This structure allows the page to support internal linking and makes the content easier to navigate.

Unlike the global css and shared html template, I centered the heading because my goal approach to creating this content page was not just go be informative and just write information, it was also a call to action for people to indeed work on their preventive health by reading the not only the information I talk about in regards to what preventive health is, but also I mean how to take the actions, thus for the h1 heading, I actually labelled it with a hint of call to action which I said preventive health: small actions, better outcomes. So centering this, will strengthen and emphasize the message and call to action I want users to understand from reading the content page, as the h1 heading is the most prominent text in all html pages.

For the images, I didn't want to overuse them and add an image element after each section to make the importance of the image seem a bit higher, as the number of images used were not a lot, hence the readers may pay attention to it more, which in this case I strategically tried to find the best images that showed people and doing something that is an example of the SDG 3, which once again promotes the call to action of the content page I want to produce.

The CSS styling weren't heavily used for this page except for the internal navigation for sections and go the top button. The internal navigation had a distinct and prominent look to it by applying border, border-left, background-color css properties to prevent the internal nav blending it too much with the background. I made the margin of the nav to be also bigger, meaning at this point the internal nav is prominent, takes up full width with a lot of vertical space, because initially without much emphasis on making the internal nav look prominent, the text links inside the nav and overall the nav didn't even looked like a navigation menu but rather just some text, but now by creating a clear distinction, the users will know that this component is nav menu for internal section because the website really emphasizes it. Lastly the go to the top button, I've used CSS technique of reducing the opacity of the background of the button so that users, when reading near the bottom of their view port / screens, they can still relatively see what the text says, which supports usability of website as well.

Accessibility

Accessibility on the content page was addressed by using meaningful headings in a logical order, descriptive alternative text for all images, and also clear internal navigation menu so users can quickly move to different parts of the page. The “Back to top” link also improves usability by allowing users to return to the beginning of the page without excessive scrolling. Section structure, readable paragraphs, and semantic HTML all help support both screen reader users and general readability. Lastly the aria-label attribute added in the nav tag for the internal nav menu for sections helps users with screen readers to be able to distinguish between the difference of the internal and the main nav menu, as I had both of them on the same page.

Challenges and Lessons Learned

One challenge was balancing the coursework requirements with page clarity. On the Feedback page, I needed to include several different sections such as the programme directory, reviews summary, and interactive form while keeping the page organised. I resolved this by separating the page into clearly defined content sections and using repeated card structures to keep the layout consistent.

Another challenge was implementing JavaScript validation in a way that gave useful feedback without making the form logic too confusing. I addressed this by separating the validation into functions, clearing previous errors before each validation attempt, and showing messages near the relevant fields. This made the code easier to read and the user feedback easier to understand.

On the Team page, one difficulty was keeping the expandable details short enough to fit inside the card layout while still saying something meaningful. I resolved this by shortening the contribution text and using CSS transitions so the details could appear without making the card look too cluttered by default.

Accessibility Considerations

Across my pages, I tried to apply accessibility through semantic structure, alternative text, readable content hierarchy, labelled form fields, and meaningful internal navigation. On interactive elements, I considered both visual users and keyboard users. On the Feedback page, I used labels, grouped fields, and a live region for submission confirmation. On the Team page, I used focusable interactive cards and visible focus styling. On the Content page, I used heading hierarchy, internal anchor navigation, and descriptive image text.

These choices improve usability by helping users understand page structure more easily, interact with forms more clearly, and access content via different methods (screen readers, clicking, typing)

Compliance

The pages I created were designed to comply with responsible web publishing and JANET-related expectations by avoiding offensive, illegal, or misleading content and by keeping the website focused on an educational topic. The content is relevant to the coursework theme and does not include material that would be inappropriate for a university-hosted student website.

I also considered responsible use of external resources. Images used on the site should be referenced properly in the relevant documentation, and any third-party resources should be used in a way that respects copyright and attribution requirements. In addition, the written content of the pages was produced for the coursework rather than copied directly from external sources, which supports academic integrity.

The pages also avoid collecting or transmitting real personal data to a server. For example, the feedback form is interactive for user experience purposes, but it does not actually submit data to a backend system. This makes it more appropriate for a static coursework website and reduces privacy risks.
